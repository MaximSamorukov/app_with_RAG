## 🔴 Критические замечания (требуют исправления до начала разработки)

### 1. Несоответствие размерности вектора +

**Проблема:** В п. 4.4 таблица `chunks`:
```sql
embedding   vector(1536),  -- размер зависит от провайдера
```

**Противоречие:** В `questions.md` указано, что pgvector **без фиксированной размерности**, а валидация на уровне сервиса. Здесь же жёстко задано `vector(1536)` — это размерность OpenAI `text-embedding-3-small`.

**Риск:** При смене провайдера (например, на Anthropic/voyage-3 с размерностью 1024) миграция потребует пересоздания колонки.

**Рекомендация:** Убрать размерность из DDL, добавить валидацию на уровне сервиса:
```sql
embedding   vector,  -- без размерности
```

---

### 2. Устаревший тип индекса +

**Проблема:** В п. 4.4:
```sql
USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
```

**Противоречие:** В `questions.md` принято решение использовать **HNSW** вместо ivfflat.

**Рекомендация:** Исправить на:
```sql
USING hnsw (embedding vector_cosine_ops);
```

---

### 3. Отсутствует таблица `settings`

**Проблема:** В п. 3.6 функционального описания упоминаются настройки AI-провайдера и webhook-уведомлений, но в схеме БД (раздел 4) нет таблицы `settings`.

**Рекомендация:** Добавить:
```sql
CREATE TABLE settings (
  key        VARCHAR(100) PRIMARY KEY,
  value      JSONB NOT NULL,
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  updated_by UUID REFERENCES users(id)
);
```

---

### 4. Нет аудита исключения чанков +

**Проблема:** В п. 4.4 таблица `chunks` не содержит полей `deactivated_by` и `deactivated_at`, хотя в `questions.md` принято решение об аудите.

**Рекомендация:** Добавить:
```sql
is_active       BOOLEAN NOT NULL DEFAULT true,
deactivated_by  UUID REFERENCES users(id),
deactivated_at  TIMESTAMPTZ,
```

---

### 5. Нет лимита сообщений в сессии +

**Проблема:** В п. 4.8 таблица `chat_sessions` не содержит поля `message_count` для быстрой проверки лимита 200 сообщений (принято в `questions.md`).

**Рекомендация:** Добавить:
```sql
message_count   INTEGER NOT NULL DEFAULT 0
```

## 🟡 Замечания средней важности +

### 6. Bull vs BullMQ

**Проблема:** В п. 3.1 указан **Bull 4.x**, в п. 2.2 — **Bull Queue**. В `questions.md` принято решение использовать **BullMQ**.

**Рекомендация:** Исправить на `BullMQ 5.x` во всех разделах.

### 9. Смешение очередей в схеме

**Проблема:** В п. 2.2 показана одна очередь, в п. 8.2 — `document-processing`. В `questions.md` принято решение о разделении на `heavy` и `light`.

**Рекомендация:** Уточнить:
```typescript
// Две очереди:
// - document-processing:heavy (PDF/DOCX, concurrency=2)
// - document-processing:light (MD/TXT, concurrency=10)
```

---

### 10. Нет ограничения на 1 активную сессию

**Проблема:** В п. 4.7 нет механизма обеспечения одной активной сессии на пользователя (принято в `questions.md`).

**Рекомендация:** Добавить поле `is_active` или триггер, который деактивирует старые сессии при создании новой. Альтернативно — логика на уровне сервиса.

---

### 11. Отсутствует cross-encoder реранкинг в пайплайне

**Проблема:** В п. 6.2 показан прямой поиск → ответ. В `questions.md` принят cross-encoder реранкинг.

**Рекомендация:** Обновить схему:
```
Vector search (top-20) → Cross-encoder rerank → top-5 в промпт
```

---

### 12. Нет endpoint'а для массового исключения чанков

**Проблема:** В п. 5.3 нет endpoint'а для массового исключения чанков (`PATCH /documents/:id/chunks/bulk`), принятого в `questions.md`.

**Рекомендация:** Добавить:
```
PATCH  /api/v1/documents/:id/chunks/bulk  — { chunkIds: UUID[], isActive: boolean }
```

## 📋 Дополнительные рекомендации

### A. Добавить soft delete для документов

Сейчас `DELETE /documents/:id` удаляет физически. Для аудита и восстановления полезно добавить:
```sql
deleted_at  TIMESTAMPTZ,
deleted_by  UUID REFERENCES users(id)
```

---

### B. Добавить триггеры для updated_at

Для таблиц `users`, `documents`, `instructions`, `chat_sessions`:
```sql
CREATE TRIGGER update_users_updated_at
  BEFORE UPDATE ON users
  FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

---

### C. Уточнить политику хранения refresh-токенов

В п. 4.2 нет TTL или политики очистки истёкших токенов.

**Рекомендация:** Добавить cron-задачу:
```sql
-- Удаление истёкших refresh-токенов раз в 24 часа
DELETE FROM refresh_tokens WHERE expires_at < NOW();
```

---

### D. Добавить индекс для поиска по сессиям

Для сайдбара истории чатов:
```sql
CREATE INDEX idx_chat_sessions_user_created 
  ON chat_sessions(user_id, created_at DESC);
```

---

### E. Уточнить формат sources в SSE

В п. 10.2 формат упрощённый. Рекомендую расширить:
```json
{
  "document_id": "uuid",
  "document_name": "...",
  "chunk_id": "uuid",
  "chunk_index": 5,
  "excerpt": "...",
  "similarity_score": 0.87
}
```
