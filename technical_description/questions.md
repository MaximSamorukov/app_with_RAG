    RAG-пайплайн
     4. Стратегия чанкинга: Какой алгоритм? Фиксированный размер по токенам, семантические границы или с учётом структуры документа
        (заголовки/параграфы)?
        Ответ: 
Гибридный подход с приоритетами:

DOCX / MD: структурный чанкинг по заголовкам и параграфам, с fallback на фиксированное окно если секция превышает 512 токенов.
PDF: фиксированное окно (512/64) как единственный надёжный вариант при отсутствии структуры из pdf-parse.
Граница чанка по возможности выравнивается по концу предложения (не обрывается на середине).

     5. Размерность эмбеддингов: Какая размерность pgvector настроена? Не фиксирует ли это систему на конкретных моделях эмбеддингов?
     Ответ: 
Столбец: vector без фиксированной размерности (pgvector 0.5+, HNSW индекс вместо ivfflat).
AI_EMBEDDING_DIMENSIONS в env — используется для валидации на уровне сервиса, не БД.
При смене провайдера — обязательная переиндексация всех документов (добавить явное предупреждение в ARCHITECTURE.md и admin UI).
Индекс: заменить ivfflat на HNSW (vector_cosine_ops) — он не требует указания размерности и показывает лучший recall.

     6. Реранкинг: Есть ли этап cross-encoder реранкинга после первоначального поиска по cosine similarity, или только векторный поиск?
     Ответ: 
     Cross-encoder реранкинг
После векторного поиска top-K чанков прогоняются через cross-encoder (например, cross-encoder/ms-marco-MiniLM-L-6-v2 локально или Cohere Rerank API). Значительно улучшает точность, но добавляет latency 200–500 мс и усложняет инфраструктуру.

     7. OCR в планах: «OCR-заглушка» — планируется ли реальная OCR (Tesseract, Azure Vision) или намеренно исключено?
     Ответ: 
Намеренно исключить OCR
Явно документировать ограничение: «сканированные PDF не поддерживаются». Детектировать пустой текст после pdf-parse → устанавливать status: 'error' с сообщением «Документ содержит только изображения, OCR не поддерживается». Честно, просто, без внешних зависимостей.

    ⚠ Слабые стороны и риски реализации

    1. Бутылочное горлышко S3 + локальная обработка

     1 Файл → S3 → Воркер → Скачивание → Парсинг → Чанкинг → Эмбеддинг → pgvector
    Риск: Каждый воркер должен скачивать файлы из S3 для обработки. Для файлов 50MB при конкурентной загрузке это создаёт I/O contention.
Ответ:
Явно задокументировать memory budget воркера: --max-old-space-size=512 для одного воркер-процесса при concurrency=5.
Разделить очереди: document-processing:heavy (PDF/DOCX, concurrency=2) и document-processing:light (MD/TXT, concurrency=10) — решает и I/O contention и даёт честный приоритет лёгким файлам.
Освобождать Buffer явно после TextExtractorService.extract — buffer = null + GC hint.
S3 и воркер обязательно в одном регионе — зафиксировать в ARCHITECTURE.md как operational requirement.
Добавить размер файла в метрики query_logs — для будущего capacity planning.

    2. Гонки при тихом обновлении токена
    Множественные параллельные API-вызовы могут одновременно запустить refresh при истечении access-токена.

Ответ:     Рекомендация: Реализовать lock на обновление токена / очередь промисов, чтобы только один запрос обновлял, а остальные ожидали его
    завершения.

    3. Деградация векторного поиска
    Не упомянуто:
     - Гибридный поиск (ключевые слова + вектор) — чистый семантический поиск пропускает точные совпадения (коды ошибок, артикулы).
     - Фильтрация по namespace — поиск только по активным документам требует фильтрации document_id IN (...), что замедляет pgvector-запросы.
Ответ: Векторный поиск + FTS (RRF)  →  фильтрация активных  →  cross-encoder реранкинг  →  top-5 в промпт
    4. Отказоустойчивость LLM-провайдера
    Одна переменная AI_PROVIDER означает отсутствие fallback, если OpenAI/Anthropic недоступен во время сессии пользователя.

    Ответ:
    Явная деградация при недоступности провайдера уже описана в UX-состояниях («LLM-провайдер недоступен → toast + предложение повторить»). Этого достаточно для MVP.
Что добавить как архитектурную заготовку без реализации:

В AIProviderFactory предусмотреть возможность принять массив провайдеров вместо одного — интерфейс create(primary, fallback?). Реализация fallback-логики — post-MVP.
В ARCHITECTURE.md явно задокументировать: «система поддерживает один активный провайдер; multi-provider failover — за рамками текущей версии».
Health check эндпоинт (GET /api/v1/health) уже пингует провайдера — этого достаточно для мониторинга недоступности на уровне ops.

    5. Пробелы UX исключения чанков
    Администратор может исключить отдельные чанки, но:
     - Нет аудита: кто, когда и какой чанк исключил.
     - Нет массовых операций (исключить все чанки из раздела документа).
     - Исключённые чанки не видны в прозрачности источников чата (пользователь не знает, что контент был намеренно исключён).
Ответ:
Добавить в таблицу chunks поля deactivated_by UUID REFERENCES users(id) и deactivated_at TIMESTAMPTZ. При PATCH /chunks/:chunkId записывать кто и когда исключил. Отображать в UI страницы чанков отдельной колонкой.
По массовым операциям:
Добавить эндпоинт PATCH /documents/:id/chunks/bulk — принимает { chunkIds: UUID[], isActive: boolean }. В UI — checkbox-выделение строк таблицы + кнопка «Исключить выбранные». Фильтрация по странице/разделу документа как опциональный параметр.
По прозрачности в чате:
Намеренно не раскрывать пользователю факт исключения чанков — это административное решение, не пользовательский контекст. Вместо этого: если ответ был дан без части потенциально релевантного контента — поведение уже корректно покрывается сообщением «Ответ не найден в базе знаний» при noContext: true. Дополнительная прозрачность здесь создаёт риск утечки структуры внутренней базы знаний.

    6. Ограничение версионирования инструкций
    «Последние 10 версий» — произвольный лимит. Для организаций с требованиями compliance может потребоваться бессрочное хранение с архивацией.
    Ответ: оставляем как есть

    7. Хранение сессий чата
    Полная история переписки может расти неограниченно. Не указано:
     - Максимальное количество сообщений в сессии.
     - Политика авто-архивации.
     - Квоты хранения на пользователя.
  Ответ:
  1 активная сессия на пользователя.
Лимит 200 сообщений на сессию.
При достижении 200 — текущая сессия удаляется, автоматически создаётся новая, пользователь получает уведомление «История очищена, начат новый диалог».
В LLM-контекст передаются последние 10 сообщений (sliding window), не вся сессия.
Счётчик сообщений — денормализованное поле message_count INTEGER DEFAULT 0 в chat_sessions для быстрой проверки без COUNT-запроса.

     - A/B-тестирование: Запуск двух инструкций параллельно на подмножестве запросов, сравнение удовлетворённости пользователей.

    F. Экспорт/резервное копирование
    Не упомянуто:

    🏗 Архитектурные проблемы


    ┌───────────────────┬────────┬─────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
    │ Проблема          │ Сер... │ Митигация                                                                                                   │
    ├───────────────────┼────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
    │
    │ **Отказы фоновых... │ Сре... │ Если воркер падает mid-processing, документ остаётся processing навсегда. Нужен timeout + job восстановл...
    │
    │ **Инъекция в шаб... │ Низкая │ Администраторы могут внедрять {{question}} в системный промпт. Валидировать/санитизировать переменные ша...
    │
    └───────────────────┴────────┴─────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

Ответ: По зависшим задачам (stuck jobs):
BullMQ имеет встроенный механизм stalledInterval — задачи в статусе active без heartbeat более N секунд автоматически возвращаются в очередь. Добавить явную конфигурацию: stalledInterval: 30000 (30 сек), maxStalledCount: 3. При превышении maxStalledCount — Document.status = 'error' с сообщением 'Worker timeout'. Дополнительно: cron-задача раз в 5 минут проверяет документы в статусе processing старше 15 минут и сбрасывает их в error.
По инъекции в шаблон промпта:
Позиция зависит от модели угрозы. Администратор — доверенная роль, намеренное внедрение вредоносного промпта это insider threat, а не внешняя атака. Однако {{question}} подставляется из пользовательского ввода — и это реальный вектор prompt injection через вопрос пользователя.
Рекомендую два уровня защиты:

Санитизация {{question}} на уровне сервиса перед подстановкой: экранировать управляющие конструкции, ограничить длину (max 1000 символов), стриппить потенциальные injection-паттерны (ignore previous instructions и подобные).
Валидация шаблона инструкции при сохранении: разрешены только переменные {{context}} и {{question}}, остальные {{...}} — ошибка валидации.
