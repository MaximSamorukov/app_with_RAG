# RAG Assistant — Описание функционала приложения

> **Тип:** SPA (Single Page Application)  
> **Стек:** React · Node.js · TypeORM · PostgreSQL · S3 · LLM (provider-agnostic)  
> **Версия документа:** 1.1 · Март 2026

---

## 1. Общая концепция

**RAG Assistant** — корпоративное веб-приложение для одной организации, позволяющее формировать собственную базу знаний и предоставлять пользователям интеллектуальный чат-интерфейс. Ответы генерируются языковой моделью (LLM) на основании релевантных фрагментов из загруженных документов — подход Retrieval-Augmented Generation (RAG).

Приложение состоит из двух функциональных зон:

| Зона | Аудитория | Назначение |
|------|-----------|------------|
| **Панель администратора** | Роль `admin` | Управление корпусом знаний и системными инструкциями |
| **Клиентский интерфейс** | Роль `user` | Задавать вопросы и получать ответы на основе RAG |

## 2. Аутентификация и управление доступом

### 2.1 Механизм аутентификации

- Вход по логину и паролю; пароли хранятся в виде bcrypt-хешей.
- После успешного входа сервер выдаёт два JWT-токена: **access token** (15 мин) и **refresh token** (7 дней).
- Refresh token хранится в HttpOnly Cookie (`SameSite=Strict`); access token — в памяти SPA (не в localStorage).
- **Одна активная сессия на пользователя**: новый логин автоматически инвалидирует все предыдущие refresh-токены.
- **Истечение refresh-токена** (7 дней неактивности): чат-сессия удаляется из БД, пользователь перенаправляется на `/login`.
- **Silent refresh**: тихое обновление access token через axios interceptor с lock на основе очереди промисов — только один реальный вызов `/auth/refresh` при параллельных запросах.
- **Самостоятельный сброс пароля**: пользователь запрашивает ссылку по email; временный токен действует 1 час.
- **CSRF-защита**: на `POST /auth/refresh` — проверка заголовка `Origin` / `Referer` против whitelist (defense-in-depth поверх `SameSite=Strict`).

## 3. Панель администратора

### 3.1 Управление документами (RAG-корпус)

#### 3.1.1 Загрузка материалов

Администратор загружает файлы через drag-and-drop или файловый диалог. Поддерживаемые форматы:

- **PDF** — извлечение текста через `pdf-parse`. Сканированные PDF (без текстового слоя) **не поддерживаются**: при обнаружении пустого текста устанавливается `status: error` с сообщением «Документ не содержит извлекаемого текста. Загрузите PDF с текстовым слоем».
- **DOCX / Word** — извлечение через `mammoth.js`; структура заголовков сохраняется для структурного чанкинга.
- **MD / TXT** — прямое чтение; Markdown конвертируется в plain text.

**Стратегия чанкинга (гибридная):**
- **DOCX / MD**: структурный чанкинг по заголовкам и параграфам; fallback на скользящее окно (512 токенов, overlap 64) если секция превышает лимит. Граница чанка выравнивается по концу предложения.
- **PDF**: только скользящее окно (512 / 64).

**Процесс после загрузки:**

1. Файл сохраняется в S3 (оригинал).
2. Фоновый воркер (BullMQ): извлечение текста → чанкинг → эмбеддинги → pgvector.
3. Статус обновляется polling'ом каждые 3 сек: `pending` → `processing` → `indexed` / `error`.
4. При завершении — опциональное webhook-уведомление.

**Лимиты:** не более 3 одновременных задач индексации на пользователя; PDF/DOCX — очередь `heavy` (concurrency 2), MD/TXT — `light` (concurrency 10).

Каждому документу задаётся: **название**, **теги**, **описание**, **флаг активности**. Версионирование документов не поддерживается — замена (v1 → v2) является ответственностью администратора.

#### 3.1.2 Список и редактирование документов

- Столбцы: название, формат, теги, статус, дата загрузки, активность.
- Фильтрация по статусу, тегам, формату; поиск по названию.
- Действия: **метаданные**, **скачать**, **переиндексировать**, **активировать/деактивировать**, **удалить** (confirm-диалог; удаляет S3 + чанки + запись).

#### 3.1.3 Просмотр и управление чанками

- Таблица чанков: текст, токены, страница, статус активности.
- **Аудит**: колонки `deactivated_by` и `deactivated_at` — кто и когда исключил чанк.
- **Индивидуальное исключение**: toggle активности.
- **Массовое исключение**: checkbox-выделение + «Исключить выбранные» (`PATCH /documents/:id/chunks/bulk`).

### 3.2 Управление инструкциями (System Prompts)

#### 3.2.1 Список инструкций

- Таблица: название, статус (активна / черновик), дата изменения.
- Одна инструкция активна одновременно; быстрое переключение из списка.

#### 3.2.2 Редактор инструкций

- CodeMirror 6 с Markdown-превью.
- Переменные `{{context}}` и `{{question}}`; любые другие `{{...}}` — ошибка валидации при сохранении.
- **Dry run**: тестовый вопрос → полный ответ с источниками без записи в логи.
- **Версионирование**: последние 10 версий с возможностью отката.

---

### 3.3 Управление пользователями

- Таблица: имя, email, роль, статус, последний вход.
- Создание с назначением роли (`admin` / `user`).
- Изменение роли, принудительный сброс пароля (письмо на email).
- Блокировка / разблокировка.

---

### 3.4 Дашборд

- Документы: всего / активных / с ошибками.
- Чанки в pgvector.
- Запросы за сегодня и 7 дней (линейный график).
- Активная инструкция со ссылкой.
- Последние 5 загруженных документов.

---

### 3.5 Журнал запросов

- Таблица: дата, пользователь, вопрос (truncated), модель, latency, `has_context`.
- Фильтры: period, пользователь, `has_context`.
- Раскрываемая строка: полный вопрос, чанки-источники с similarity score.
- **Экспорт CSV**: `GET /analytics/query-logs/export` с теми же фильтрами. Поля: дата, email, вопрос, модель, latency_ms, has_context, similarity_scores.

---

### 3.6 Настройки

**AI-провайдер**: выбор провайдера и моделей completion/embedding, регулировка temperature, тест соединения. Конкретная модель не хардкодится. При смене провайдера с другой размерностью эмбеддингов — предупреждение о необходимости переиндексации.

**Webhook-уведомления** (post-MVP, feature-flag): `webhook_url` + `webhook_events` (`document.indexed`, `document.error`). Воркер делает POST после завершения задачи.

## 4. Клиентский интерфейс

### 4.1 Чат

- Enter — отправка, Shift+Enter — перенос строки.
- Ответ стримится (SSE), рендерится через `react-markdown`.
- Typing indicator до первого токена.
- Блок **«Источники»** под ответом: документы и чанки с раскрываемым текстом.
- При отсутствии релевантных фрагментов — **«Ответ не найден в базе знаний»**.

### 4.2 История диалогов

- **1 активная сессия** на пользователя.
- Лимит **200 сообщений**. При достижении — сессия удаляется, создаётся новая, уведомление: «История очищена, начат новый диалог».
- В LLM-контекст передаются последние **10 сообщений** (sliding window).
- При истечении refresh-токена — сессия удаляется.

### 4.3 Настройки пользователя

- Смена пароля; самостоятельный сброс по email.
- Тема (светлая / тёмная).

## 5. Сквозная функциональность

### 5.1 Логирование запросов

Каждый RAG-запрос: вопрос, ID чанков + similarity, провайдер, модель, latency, `has_context`. Хранение 5 дней.

### 5.2 Provider-agnostic AI адаптер

```typescript
interface AIProvider {
  generateEmbedding(text: string): Promise<number[]>
  generateCompletion(messages: Message[], options: CompletionOptions): AsyncIterable<string>
}
```

`AIProviderFactory.create(primary, fallback?)` — интерфейс предусматривает fallback (реализация post-MVP). Система поддерживает один активный провайдер; multi-provider failover — за рамками текущей версии.

### 5.3 Обработка ошибок и UX-состояния

| Состояние | Отображение |
|-----------|-------------|
| Документ обрабатывается | Прогресс + `indexing...` |
| Сканированный PDF | `error`: «Документ не содержит извлекаемого текста» |
| Ошибка индексации | `error` + причина + «Повторить» |
| Воркер завис | `error`: «Worker timeout» (автосброс через 15 мин) |
| LLM недоступен | Toast + «Попробуйте позже» |
| Нет контекста | «Ответ не найден в базе знаний» |
| Лимит 200 сообщений | «История очищена, начат новый диалог» |
| Истечение токена | Редирект `/login`, сессия удалена |
| Доступ без прав | user → `/chat`; неавторизован → `/login` |

## 6. Нефункциональные требования

| Требование | Целевой показатель |
|------------|--------------------|
| Время ответа API p95 (без LLM) | < 200 мс |
| Начало стриминга ответа | < 1 с |
| Максимальный размер файла | 50 МБ |
| Concurrency воркера | 5 глобально (heavy: 2, light: 10) |
| Параллельных сессий на пользователя | 1 |
| Сообщений в сессии | 200 (затем авто-пересоздание) |
| TTL refresh-токена | 7 дней; по истечении — сессия удаляется |
| Хранение query logs | 5 дней |
| Браузеры | Chrome/Edge/Firefox/Safari, последние 2 версии |
| Мобильная адаптация | Responsive, от 325 px |
